#

## 一、重构目标

现在正式进入 **a-top10 V2 重构实施清单编写与落地阶段**。

本阶段目标不是继续讨论抽象方案，而是把 V2 主链重构拆成 **可直接执行的开发任务清单**，用于后续按文件逐个改造、逐个验收、逐步闭环。

本轮先处理第一阶段最危险、最直接污染 Top10 终值的两段主链：

- `a_top10/steps/step5_ml_probability.py`
- `a_top10/steps/step6_final_topn.py`

第一阶段的核心任务只有一个：

> **先把“概率语义”与“最终排序”从旧链污染中剥离出来，形成干净、可解释、可兼容旧产物的 V2 终排主链。**

## 二、第一阶段必须遵守的硬约束

### 约束 1：外部产物契约不变

所有对外产物的 **文件形式、文件名、落库路径、消费链入口** 必须保持兼容。

尤其以下约束不可破坏：

- `outputs/decisio/` 路径不改
- 原有对外文件名尽量不改
- 原有下游读取逻辑不断链
- 原有报表/页面/消费端还能继续读取

### 约束 2：不新建平行 V2 文件树

本轮不建议另外新建整套 `step5_v2.py / step6_v2.py` 平行分支长期共存。

原则上优先在原文件骨架中完成重构，必要时只允许新增：

- 小型 helper 函数
- 同目录下辅助模块
- 配置项
- 兼容字段转换层

但不能把主链拆成另一套长期并行体系。

### 约束 3：Step6 必须成为纯终排器

`step6_final_topn.py` 以后必须只负责：

- 接收候选输入
- 接收概率输入
- 计算最终综合分
- 产出 Top10 排序
- 产出剩余全部排序
- 按旧契约落库

不得继续承担过多“补字段、补报告、兼容兜底、回填临时逻辑、杂项写出”的污染职责。

### 约束 4：预测报告必须固定输出两部分

V2 主链重构后，对外认知口径固定为：

- **Top10 最终排序**
- **剩余全部排序**

不能再只给 Top10 而丢掉全量排序。

# 1. a\_top10/steps/step5\_ml\_probability.py

**优先级：P0**\
**性质：第一阶段核心改造文件**\
**目标：把“脏 Probability”拆成可解释概率结构**

## 1.1 当前问题

当前 `step5_ml_probability.py` 的核心问题不是单纯模型效果，而是 **语义污染**：

- rule 打分、伪概率、ML 概率混在一起
- 外部看到一个 `Probability` 字段，但内部来源不纯
- 无法区分“规则先验”和“模型后验”
- Step6 只能吃到一个混杂终值，导致最终排序不可解释

## 1.2 本文件重构目标

本文件重构后，必须显式输出至少三层概率语义：

- `prob_rule`
- `prob_ml`
- `prob_final`

并保留兼容字段：

- `Probability`

其中兼容规则应明确为：

> **旧字段 ****`Probability`**** = 新字段 ****`prob_final`**

这样旧消费链不断，但内部语义已经完成净化。

## 1.3 本文件必须完成的具体改动点

### 改动点 A：明确输入口径

Step5 的输入必须明确来自前链候选结果，至少包含：

- 候选基础信息（代码、名称、交易日等）
- Step3 结果（如 `StrengthScore`）
- Step4 结果（如 `ThemeBoost`，若已有）
- 其它当前已存在且可稳定获取的特征

这里的重点不是扩字段，而是先把 **输入字段口径固定下来**，避免 Step5 内部再偷偷补各种来源不明的分数。

### 改动点 B：拆分三层概率结构

Step5 内部必须形成三层独立变量：

#### 1）`prob_rule`

规则概率层。\
含义：在 ML 缺失、样本不足、模型不可用时，根据稳定规则特征构造的先验概率。

作用：

- 兜底
- 冷启动
- 模型缺失时维持系统可运行
- 保证旧链不断

#### 2）`prob_ml`

模型概率层。\
含义：由 LR / LGBM / 其它学习器输出的标准化概率结果。

要求：

- 没模型时允许为空
- 样本不足时允许为空
- 不得强行伪造为与 `prob_rule` 同值
- 必须能追踪来源

#### 3）`prob_final`

最终概率层。\
含义：供 Step6 使用的 V2 主概率字段。

要求：

- 有模型时：以 `prob_ml` 为主，按配置决定是否融合 `prob_rule`
- 无模型时：退回 `prob_rule`
- 必须是唯一进入 Step6 的正式概率口径

### 改动点 C：新增概率来源标记

必须新增概率来源或融合标记字段，例如：

- `prob_src`
- `prob_ml_available`
- `prob_fusion_mode`

至少要让外部能看清：

- 当前记录是规则概率
- 还是模型概率
- 还是规则+模型融合结果

这一步非常重要，因为它决定后续 Step7 学习闭环是否还能学清楚。

### 改动点 D：保留旧兼容字段

为保证旧消费链不断，Step5 输出中保留：

- `Probability`

但内部明确改为：

- `Probability = prob_final`

也就是说，旧字段只是 **兼容别名**，不再是内部认知主字段。

### 改动点 E：统一概率数值范围与缺失逻辑

必须统一以下规则：

- 概率范围固定在 `[0, 1]`
- 缺失不得出现字符串脏值
- 非法值必须截断或清洗
- 模型失效时不能把异常值直接抛给 Step6

### 改动点 F：模型加载逻辑净化

当前模型加载逻辑如果存在“能加载就混一点，不能加载就伪造一点”的旧逻辑，必须重构为显式分支：

- 模型存在且可推理 → 产出 `prob_ml`
- 模型缺失 / 推理失败 → `prob_ml = null`
- 最终由 `prob_final` 统一收口

### 改动点 G：输出字段顺序与兼容整理

Step5 输出结果表中，建议至少稳定包含：

- 基础标识字段
- `StrengthScore`
- `ThemeBoost`
- `prob_rule`
- `prob_ml`
- `prob_final`
- `Probability`
- `prob_src`

这样后续 Step6 和 Step7 都能接得干净。

## 1.4 本文件验收标准

Step5 改完后，必须满足以下验收条件：

1. 不再只有一个语义混乱的 `Probability`
2. 结果中可清晰看到 `prob_rule / prob_ml / prob_final`
3. 旧字段 `Probability` 仍存在
4. 主程序不崩
5. Step6 可以直接只吃 `prob_final`
6. 无模型情况下仍能跑通

# 2. a\_top10/steps/step6\_final\_topn.py

**优先级：P0**\
**性质：第一阶段核心改造文件**\
**目标：把 Step6 净化为纯终排器**

## 2.1 当前问题

当前 `step6_final_topn.py` 的问题最严重，不是因为“排序公式不够好”，而是因为它承担了过多非终排职责：

- 终排
- 字段补丁
- 兼容回填
- 报表辅助
- 结果裁切
- 产物写出耦合
- 各种旧链补救逻辑

这会导致：

- 最终分数来源不清
- 排序逻辑不可解释
- 修改一处，外链全动
- 未来 Step7 也学不干净

## 2.2 本文件重构目标

Step6 重构后必须变成：

> **只负责“接收候选 + 接收 V2 概率 + 计算 final\_score + 排序 + 输出 Top10 和全量排序”。**

除此之外的职责都应该下沉到：

- 配置层
- writer 层
- 主程序入口层
- 兼容层

## 2.3 本文件必须完成的具体改动点

### 改动点 A：明确 Step6 输入字段契约

Step6 接收的数据至少要明确依赖：

- 基础标识字段
- `StrengthScore`
- `ThemeBoost`
- `prob_final`

可选兼容读取：

- `Probability`（仅兼容旧链，不作为新主字段）
- 其它旧字段

但内部正式逻辑必须优先读取：

- `prob_final`

### 改动点 B：建立唯一终排分数字段

必须显式建立一个新的最终排序分数字段，例如：

- `final_score`

以后 Step6 的核心就是：

> 用明确公式，把 `StrengthScore / ThemeBoost / prob_final` 组合为唯一终排值。

而不能再同时混出多个彼此重叠、语义不明的终排口径。

### 改动点 C：终排公式配置化

终排公式不能硬编码死在乱逻辑里，必须配置化，至少让以下权重进入配置：

- `w_prob`
- `w_strength`
- `w_theme`

默认可以先由 `prob_final` 主导，但必须允许后续在 `default.yml` 中调参。

### 改动点 D：排序输出分成两层

Step6 必须同时产出两份逻辑结果：

#### 1）Top10 最终排序

即真正对外主输出的前 10。

#### 2）剩余全部排序

即 Top10 之外的全量候选排序结果。

注意：\
这不是“可选增强”，而是 V2 固定输出要求。

### 改动点 E：去掉非终排杂责

Step6 内部必须剥离以下污染职责：

- 报表文本生成
- 额外写出多种非终排解释内容
- 乱补兼容字段
- 杂项回填
- 与 writer 深耦合的落库细节
- 无关的 fallback 拼装逻辑

Step6 只生成：

- 排序表
- Top10 子集
- 必要兼容字段

### 改动点 F：保留旧产物列兼容

虽然内部终排逻辑要净化，但对外表结构要尽量兼容。

也就是说：

- 旧消费方需要的核心字段尽量保留
- 旧字段若还存在，可改成从 `final_score` 或 `prob_final` 映射而来
- 不能因为重构导致 `outputs/decisio/` 下游直接断掉

### 改动点 G：排序稳定性规则明确

必须规定：

- 主排序键：`final_score`
- 次排序键：`prob_final`
- 再次排序键：可配置或使用稳定字段（如 `StrengthScore` / `ts_code`）

这样后续多次跑同日数据时，排序不会因为隐式 DataFrame 顺序漂移。

### 改动点 H：输出字段固定化

Step6 输出结果建议固定包含：

- rank
- ts\_code
- name
- StrengthScore
- ThemeBoost
- prob\_rule
- prob\_ml
- prob\_final
- Probability
- final\_score

至少保证对内可解释，对外可兼容。

## 2.4 本文件验收标准

Step6 改完后，必须满足：

1. 已成为纯终排器
2. 终排唯一主分数字段是 `final_score`
3. 正式概率读取口径是 `prob_final`
4. 同时输出 Top10 与剩余全部排序
5. 原有 `outputs/decisio/` 消费链不断
6. 排序规则稳定、可解释
